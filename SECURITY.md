# Security Policy

## Overview

This document describes the security policy for the AI Agent Assessment Project — a framework for evaluating, scoring, and governing AI agents before and after deployment. Because this project directly influences whether AI agents are permitted to operate, security of the assessment process itself is critical.

---

## Supported Versions

| Component | Version | Supported |
|---|---|---|
| `assess_agent.py` (CLI tool) | 1.x | ✅ Active |
| `agent_assessment_3d/` (3D kit files) | 1.x | ✅ Active |
| Code of Conduct | 1.0 | ✅ Active |
| Older assessment scripts (pre-1.0) | < 1.0 | ❌ Not supported |

Security fixes will only be applied to the current major version. If you are using an older version of the assessment tooling, upgrade before reporting issues.

## Reporting a Vulnerability

If you discover a security vulnerability in this project — including flaws in the assessment logic, scoring algorithm, logging tag generation, or documentation that could lead to unsafe agent deployment — please report it responsibly.

**Do not open a public GitHub issue for security vulnerabilities.**

Instead, report by email to: **github1979@gmail.com**

Include the following in your report:

- A clear description of the vulnerability and its potential impact
- Steps to reproduce the issue
- The component affected (`assess_agent.py`, a specific section scorer, the JSON tag output, etc.)
- Any suggested mitigations you have identified

You will receive an acknowledgement within **5 business days**. If the vulnerability is confirmed, a fix will be prioritised based on severity and a patched version released within **30 days** where possible. You will be credited in the release notes unless you request anonymity.

## Security Considerations for AI Agent Assessment

The assessment process governs whether AI agents are permitted to operate. The following risks are specific to this project and must be taken seriously by all contributors.

### 1. Score Integrity

The scoring algorithm in `assess_agent.py` is the authoritative basis for deployment decisions. Any modification to section weights, score thresholds, or the weighted average calculation must be reviewed and agreed upon by the project lead before merging. Unauthorised changes to scoring logic — even minor ones — could result in under-qualified agents receiving a GREEN verdict.

**Mitigation:** All changes to `assess_agent.py` require a pull request with at least one independent reviewer. The scoring weights in `WEIGHTS` and the thresholds (GREEN ≥ 7, AMBER 4–6, RED ≤ 3) are considered security-sensitive constants.

### 2. Assessment Report Tampering

Generated markdown reports (`[agent-name].md`) and JSON tag files (`[agent-name]-tags.json`) are the authoritative records of an assessment. These files must not be edited after generation. Altering a report to improve a score or suppress attention points undermines the governance process.

**Mitigation:** Assessment reports should be committed to version control immediately after generation and treated as immutable audit records. Any re-assessment requires running the tool fresh, not editing the output.

### 3. Logging Tag Bypass

The JSON logging tags generated by the assessment tool (including `agent_start`, `agent_stop`, `tokens_used`, `agent_error`) are a required part of agent deployment. Agents that are deployed without these tags attached to their observability pipeline are operating outside the governance framework.

**Mitigation:** Deployment checklists must include verification that logging tags are active before an agent is permitted to run in production.

### 4. Data Sensitivity Misclassification

Section 4 of the assessment (Data Sensitivity) requires honest classification of the data the agent will access. Intentionally or carelessly misclassifying sensitive data (e.g., marking Restricted data as Internal to avoid stricter controls) creates a material security risk.

**Mitigation:** Assessors independent of the agent owner must review the Data Sensitivity section for any agent handling Confidential or Restricted data. PII handling procedures must be verified before a GREEN verdict is issued for such agents.

### 5. Scope Creep After Assessment

An agent that receives approval based on a defined scope and then quietly expands its access, data sources, or actions without a re-assessment represents a governance failure with security implications.

**Mitigation:** Any material change to an agent's purpose, task list, data access, or tool permissions requires a full re-assessment before the change is deployed. The Code of Conduct makes this a mandatory obligation of the agent owner.

### 6. Physical Assessment Kit Integrity

The 3D-printed physical tokens (tiles, chips, toppers, dial, flags) are used in collaborative assessment sessions. The integrity of those sessions depends on honest placement of scoring chips and attention flags.

**Mitigation:** Assessment sessions must include at least one assessor who is independent of the agent owner. Session outcomes must be recorded digitally using the CLI tool — physical token placement alone is not a sufficient record.

## Sensitive Data in This Repository

This repository does not intentionally store secrets, credentials, or personal data. However, generated assessment reports may contain agent owner names, team names, model identifiers, and cost figures. Review generated files before committing them if any of this information is sensitive in your context.

The `.gitignore` already excludes `.env` files and raw data directories. Do not commit API keys, access tokens, or credentials of any kind.

## Dependency Security

This project uses the following external dependencies. Keep them updated:

| Dependency | Used for | Check for updates |
|---|---|---|
| `trimesh` | 3D STL generation | `pip install --upgrade trimesh` |
| `manifold3d` | Boolean mesh operations | `pip install --upgrade manifold3d` |
| `shapely` | 2D polygon geometry | `pip install --upgrade shapely` |
| `Pillow` | Bitmap text rendering | `pip install --upgrade Pillow` |
| `numpy` | Numerical operations | `pip install --upgrade numpy` |
| `docx` (npm) | Word document generation | `npm update docx` |

Run `pip list --outdated` and `npm outdated` periodically to catch vulnerabilities in dependencies.

## Responsible Disclosure

This project operates under a responsible disclosure policy. We ask that:

- You give us a reasonable opportunity to investigate and fix the issue before public disclosure.
- You do not exploit the vulnerability beyond what is necessary to demonstrate it.
- You act in good faith throughout the disclosure process.

We commit to responding promptly, keeping you informed of progress, and not pursuing legal action against researchers acting in good faith.
